# 4.2 测试缺陷与修正过程

## 缺陷一：批量账单写入过程中界面卡顿

【缺陷描述】  
在早期版本中，执行批量导入或高频新增账单时，应用首页与统计页出现明显卡顿。实测在连续插入 5,000 条账单的过程中，列表滚动掉帧，局部场景出现 1 s 以上界面无响应，影响批量记账与测试数据构造效率。

【原因分析】  
排查发现，早期实现采用逐条插入方式直接调用 DAO 写库，每次插入均独立开启提交事务，导致 SQLite 提交频率过高，磁盘同步开销集中释放；同时，账单写入后立即触发列表刷新与统计重算，造成 UI 线程与数据库写线程竞争资源。在默认日志模式下，频繁提交进一步放大了 I/O 等待时间，最终表现为界面卡顿。

【修正过程】  
针对写入路径进行两项优化。其一，将逐条插入重构为批量写入流程，统一使用 `BatchQueryHelper.bulkInsert` 合并 SQL 执行批次，减少事务提交次数与 ArkTS 层到数据库层的调用开销。其二，在数据库初始化阶段启用 `PRAGMA journal_mode=WAL`，使读写操作可并发进行，降低高频写入对查询与界面刷新的阻塞影响。优化后同步梳理了插入完成后的页面刷新逻辑，将非必要统计更新延后到批次提交完成后统一触发，避免 UI 线程频繁重绘。

【复测结果】  
在相同设备和数据规模下复测 5,000 条账单批量写入任务，写入总耗时由 8.6 s 降至 2.1 s，性能提升约 75.6%；主界面滚动帧率由约 24 FPS 提升至 53 FPS，未再出现长时间无响应现象。扩展至 10,000 条并发事务写入场景后，系统仍可稳定完成提交，与 4.1 节并发事务写入测试结果一致。

## 缺陷二：大文件加密导出阶段的内存占用异常

【缺陷描述】  
在早期导出实现中，当用户导出包含大量账单及关联配置的数据文件时，AES-256-GCM 加密阶段出现内存占用快速上升。测试中，当导出文件接近 4 MB 且连续执行多次导出操作时，个别设备出现内存峰值过高、导出失败，少数情况下还伴随导入端校验不一致的问题。

【原因分析】  
问题根因在于初版实现采用整文件一次性读入内存、序列化、加密再整体写出的处理方式，导致明文缓冲区、密文缓冲区和序列化中间对象在同一时段叠加驻留，放大了瞬时内存压力。同时，早期导出流程缺乏端到端完整性校验，一旦文件在写入、中转或同步过程中发生截断或局部篡改，仅依赖解密阶段报错，难以及时识别具体数据一致性问题。

【修正过程】  
首先将导出流程改造为分阶段、分块处理模式，避免一次性持有完整明文与密文数据，降低加密过程中的峰值内存占用。其次，在导出文件元数据中引入校验和机制，使用 `ChecksumHelper` 对导出内容生成摘要，并在导入与同步前后执行一致性校验；若发现摘要不一致，则立即终止后续写入并返回校验失败结果。通过该方案同时解决了大文件处理稳定性与传输链路完整性验证问题。

【复测结果】  
优化后，对包含 12,486 条账单的数据集连续执行 30 轮导出与回导测试，导出过程未再出现内存溢出或异常中断；内存峰值较早期实现下降约 41.3%。在错误口令、篡改密文和篡改认证标签场景下，系统均能稳定识别异常；对人为截断和摘要不一致文件的导入请求，校验阶段即可准确拦截，结果与 4.1 节 AES-256-GCM 加密解密测试及云端增量同步测试保持一致。
