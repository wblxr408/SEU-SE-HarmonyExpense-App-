# 二、开场话术（移动应用 + 鸿蒙）

**推荐口径（可照读缩写）：**

> HarmonyExpense 是一款面向 **HarmonyOS** 的 **移动端个人财务管理应用**。我们使用 **ArkTS** 与 **声明式 ArkUI** 开发，数据主要 **端侧持久化（RelationalStore / SQLite）**，覆盖从记账、分类与标签、统计报表，到预算与提醒、备份与云同步的完整闭环。  
> 与常见「只记账」应用相比，我们的亮点是把 **预算** 做成 **可计算、可解释、可因人而异** 的能力：**多模型消费预测 + 用户画像调参 + 个性化阈值与建议反馈**，让用户在手机上获得接近「财务助手」的体验，而不是静态数字。

**可强调的移动场景：** 高频、碎片化、弱网/离线仍要能用；因此端侧库表、缓存、分页、批量操作都是合理选型，不是堆砌名词。

## 1. 整体协调

**要讲清的「协调」有三层：用户体验协调、架构协调、数据协调。**

**（1）用户体验与信息架构**

- 主路径清晰：**记账 → 看结构（分类/标签）→ 看趋势（统计）→ 管目标（预算）→ 保安全（备份/同步）**。  
- 财务类应用忌讳「为了功能堆菜单」：你们用模块化页面 + 业务服务收口复杂逻辑，有利于保持导航和心智模型一致。

**（2）软件架构协调（与 README 一致）**

- **四层**：表示层 `pages/` → 业务层 `service/` → 数据访问 `dao/` → 基础设施 `database/` + `model/`。  
- **依赖单向**：页面不直接写复杂 SQL；DAO 只做表级 CRUD；跨表与规则在 Service。  
- **设计原则（可点名）**：单一职责（SRP）、依赖倒置（统一通过 `DatabaseManager`）、开闭原则（扩展 DAO/服务而非改核心）。

**（3）数据与模块协调**

- 外键、CHECK、UNIQUE、事务（README「安全机制」）保证账单-账户-分类-预算之间关系一致。  
- 智能预算依赖账单与分类历史：在陈述里说一句「预算不是孤立模块，而是吃真实记账数据」。

**演示建议：** 从首页进入记账 → 同一用户下看分类统计 → 进入预算/智能预算页，强调「同一套用户与数据上下文」。

**追问预案：**  
- *「智能预算数据从哪来？」* → 历史账单按分类与时间聚合，由 Service 拉取并预测，结果再写回预算/展示层。  
- *「为什么分层？」* → 移动端迭代快，分层降低页面膨胀和回归成本。

## 3. 运行流畅（性能与交互）

**口径：流畅 = 数据库快 + 应用层少重复劳动 + 列表可扩展 + 可观测。**

**（1）数据库层（README「性能优化」——可挑重点背）**

- **WAL**：`PRAGMA journal_mode=WAL`，提升并发读写、减少锁竞争。  
- **索引**：复合索引、覆盖索引、部分索引（如按日期、统计场景、未删除过滤），针对账单列表与统计查询。  
- **配置**：`synchronous=NORMAL`、`cache_size`、`temp_store=MEMORY`、`foreign_keys=ON` 等，平衡安全与速度。

**（2）应用层**

- **CacheManager**：如分类列表按 `userId` 缓存、TTL（README 示例）。  
- **BatchQueryHelper**：批量插入，减少逐条 IO。  
- **分页**：`pageSize` + **游标**（cursor），适合长账单列表，避免一次性加载。

**（3）可观测**

- **PerformanceMonitor.measure**：对关键查询打点，答辩可说「我们有性能报告接口，便于验收与优化」。

**（4）与「智能预算」的关系**

- 预测与建议计算放在 **Service**，避免在 ArkUI 构建函数里做重计算导致掉帧；结果一次性交给界面渲染。

**演示建议：** 长列表滑动、打开统计页、生成一次智能预算，口播「背后有索引/缓存/分页支撑」。

**追问预案：**  
- *「最慢的环节可能在哪？」* → 全量历史扫描与复杂预测；对策是限制回溯月数、异步、必要时进度提示（按你们实现实说）。  
- *「离线能用吗？」* → 核心记账与本地库可用；云同步依赖网络（与 README 一致）。

## 4. 创意新颖（个性化预算优化——核心卖点）

建议用 **「问题 → 方案 → 技术链路 → 个性化 → 可进化」** 五段式讲透。

**（1）问题**

- 传统预算：用户填一个数，超了就红字，**没有结合历史行为与未来波动**。  
- 用户需求：希望「帮我定一个合理预算」「哪类该紧哪类可松」「别总误报」。

**（2）方案概览**

- **智能预算计划**：基于历史消费，生成分类级分配、预测区间、建议与季节性因子、健康度等（`SmartBudgetService` 头注释与流程）。  
- **个性化层**：同样算法，**不同用户不同权重与阈值**；建议经 **历史反馈** 过滤，减少骚扰。

**（3）算法与预测（多模型 —— 体现技术含量）**

- 多种时间序列思路：**简单/加权移动平均、指数平滑、Holt-Winters、线性回归**，并支持 **集成（ensemble）** 思路，避免单一模型失真。  
- **配置化**：`ForecastEngineConfig`（算法、置信水平、窗口等）便于调参与扩展。

**（4）与用户画像联动（代码级事实）**

- `PersonalizedBudgetService.getPersonalizedWeights`：  
  - **储蓄型**：提高 SMA/WMA 权重（消费更稳定）。  
  - **消费型**：提高 ES/HW 权重（波动大、更需平滑与季节/趋势）。  
  - **年轻用户**：略增 LR/HW，略减 SMA/WMA（模式变化快）。  
  - **严格执行型**：略增简单方法权重。  
  - 最后做**权重归一化**。  
- `SmartBudgetService` 中根据 `spendingPersonality` 调整指数平滑 **alpha**（消费型更敏感、储蓄型更平滑）。

**（5）个性化阈值与建议**

- `PersonalizedThresholds`：**增/减预算建议阈值、预警阈值、置信水平**，与默认「一刀切」规则区分。  
- 生成建议时使用个性化阈值；再经 `filterAdvicesByHistory` 做**基于历史反馈的过滤**（接受/拒绝类反馈），体现学习闭环。  
- **优化建议接口** `getBudgetOptimization`：可只返回高优先级优化/预警类建议，适合移动端「少而精」的展示。

**（6）增值点（有则讲，无则略）**

- **季节性因子**检测、**预算健康度评分**（`evaluateBudgetHealth`）——作为「不仅预测，还做健康诊断」的补充。

**演示建议：**  
打开智能预算页 → 展示「分类分配/预测/建议」→ 口播「同一历史数据，储蓄型与消费型权重不同」→ 若界面有反馈入口，演示一条建议被采纳/忽略后的差异（按实际 UI）。

**追问预案：**  
- *「准吗？」* → 依赖数据量与稳定性；用置信水平、多模型集成、阈值控制「宁可给区间与提示，不乱给绝对结论」。  
- *「个性化会不会过度拟合？」* → 有权重的归一化、有默认配置、有反馈过滤而非无界记忆；可说明后续可加冷却期或最小样本量（若已实现可实说）。  
- *「和机器学习模型比？」* → 端侧轻量、可解释、无训练部署成本；适合课程/产品阶段的移动场景。
