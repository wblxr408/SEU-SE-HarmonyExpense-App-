### 第2页：架构风格与设计原则

**标题**: 四层分层架构 (Layered Architecture)

**内容要点**:

```
架构风格: 经典分层架构 (Layered Architecture)

四层架构设计
┌──────────────────────────────────────┐
│  表示层 (Presentation Layer)          │  ← 18个页面 + 2个组件
│  pages/ components/                  │
└──────────────────────────────────────┘
            ↓ 单向依赖
┌──────────────────────────────────────┐
│  业务逻辑层 (Business Logic Layer)     │  ← 13个服务 + export模块
│  service/                            │     (12,487 LOC)
└──────────────────────────────────────┘
            ↓ 单向依赖
┌──────────────────────────────────────┐
│  数据访问层 (Data Access Layer)        │  ← 16个DAO对象
│  dao/                                │     (7,547 LOC)
└──────────────────────────────────────┘
            ↓ 单向依赖
┌──────────────────────────────────────┐
│  基础设施层 (Infrastructure Layer)     │  ← 数据库管理 + 23个数据模型
│  database/ model/                    │     (10,252 LOC)
└──────────────────────────────────────┘

核心设计原则
✓ 单一职责原则 (SRP) - 每层职责清晰，DAO只负责CRUD，Service处理业务逻辑
✓ 依赖倒置原则 (DIP) - 上层依赖下层抽象，统一通过DatabaseManager获取数据库
✓ 开闭原则 (OCP) - 扩展开放修改封闭，新增功能无需修改现有代码
✓ 接口隔离原则 (ISP) - 精简接口，避免冗余依赖
```

### 第7页：数据安全与加密机制

**标题**: 数据安全 - 多层防护体系

**内容要点**:

```
数据安全架构

1. 加密传输 (EncryptionModule)
   算法: AES-256-GCM (认证加密)
   • 密钥长度: 256位
   • 加密模式: GCM (Galois/Counter Mode)
   • 认证标签: 防篡改检测
   • 密钥派生: PBKDF2
   • 迭代次数: 100,000次
   • 盐值: 随机生成16字节
   • IV: 每次加密随机生成12字节

2. 密码安全
   • 存储: SHA-256哈希
   • 验证: 哈希值比对，不存储明文
   • 会话: 24小时超时机制
   • 令牌: 随机生成唯一会话令牌

3. 数据完整性保护
   • 外键约束: 强制关系完整性
   • CHECK约束: 枚举值验证
   • UNIQUE约束: 唯一性保证(邮箱等)
   • 事务保证: ACID原子性
   • 数据哈希: SHA-256校验和验证
   • 软删除: 逻辑删除，数据可恢复

4. SQL注入防护
   • 参数化查询: 使用PreparedStatement
   • 严格类型检查: TypeScript强类型
   • 输入验证: Model层validate方法
   • 白名单验证: 枚举值限制

5. 用户数据隔离
   • 多租户隔离: 所有表都有userId字段
   • 查询过滤: 所有查询强制带userId条件
   • 权限验证: 会话级别权限控制

6. 备份与恢复
   • 加密备份: 支持密码保护的备份
   • 完整性校验: ChecksumHelper验证
   • 版本控制: 备份文件包含版本信息
   • 增量备份: 仅备份变更数据

安全等级
┌─────────────────────────────────────────┐
│ 传输层: AES-256-GCM + PBKDF2            │
│ 存储层: 软删除 + 外键约束 + 哈希校验     │
│ 访问层: 用户隔离 + 会话超时 + 权限控制   │
│ 应用层: 参数化查询 + 类型检查 + 输入验证 │
└─────────────────────────────────────────┘

符合标准
✓ OWASP Top 10 安全实践
✓ 数据加密传输与存储
✓ SQL注入防护
✓ XSS防护 (UI层输入过滤)
✓ GDPR数据保护合规
```

## 答辩可能问题准备

### 架构相关

Q: 为什么选择四层架构而不是MVC或MVVM?
A: 四层架构职责更清晰，业务逻辑层和数据访问层分离，便于大型项目维护和团队协作。

Q: 如何保证跨层调用的性能?
A: 通过缓存层、批量操作、索引优化等多层优化策略，同时使用性能监控实时跟踪。

### 数据库相关

Q: 为什么使用软删除而不是物理删除?
A: 软删除可以保留数据历史，支持数据恢复，满足审计需求，更安全可靠。

Q: 如何处理大数据量的查询性能问题?
A: 通过索引优化(复合/部分/覆盖索引)、分页查询、缓存策略、批量操作等多种手段优化。

### 安全相关

Q: 如何防止SQL注入攻击?
A: 使用参数化查询、严格类型检查、输入验证三重防护。

Q: 数据加密的性能开销如何平衡?
A: 仅对敏感数据和传输数据加密，本地存储使用软删除和权限控制，按需加密。

### 功能相关

Q: 云端同步如何处理冲突?
A: 使用时间戳比较、哈希值验证、版本号控制，预留手动解决接口。

Q: AI智能助手的实现原理?
A: 通过LLMService对接大模型API，结合用户上下文和历史数据生成个性化建议。
